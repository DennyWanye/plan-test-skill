# Phase 1 — 写 plan

**目的**：基于 `{ARCH_DIR}/ARCHITECTURE.md` 与 `{ACCEPTANCE_FILE}`，写出一份**代码级可执行**的优化/升级 plan。

## 步骤

1. **最佳实践调研**（不可跳过）
   - 针对需求里的每个问题，调研业界最佳实践该怎么解决。
   - 把"为什么选这个方案、放弃了哪些备选、权衡是什么"写进 plan。

2. **写 plan，细到代码级**
   - 存放：`{PLANS_DIR}/<按需求命名的新文件夹>/plan.md`。
   - 每个改动点必须明确：**改哪个文件 → 哪段/哪个函数 → 怎么改 → 改成什么样**。
   - 把 plan 切成可独立执行、可独立验证的"咬一口大小"的任务。
   - 每个任务标注它满足 `{ACCEPTANCE_FILE}` 里的哪条 AC（建立可追溯链的起点）。

3. **铁律**
   - `FEATURE_POLICY = only-add`：功能只能多做，不可少做。
   - 必须含**实现细节调研**结论，不只是"做什么"。

## plan.md 建议结构

```markdown
# Plan：<需求名>

## 关联验收标准
- 覆盖 AC-1, AC-2, …（每条都要有任务覆盖）

## 文件影响清单
| 文件 | 职责 | 本次改动 |
|------|------|----------|

## 任务清单（按依赖排序）
### Task 1 — <标题>  [覆盖 AC-1]
- 改动文件：path/to/file
- 现状：（已读过的当前实现）
- 修改方式：（代码级，怎么改）
- 验证：（怎么测这一步）
- 依赖：Task X

### Task 2 …
```

## 出口

- plan 覆盖全部"必须" AC、每个任务到代码级 → 进入 phase-2 迭代。
