# 幂等性审查清单

看到"遍历 + 写副作用"的代码，逐条追问（任何一条答不上来 = 风险点，必须补测/补逻辑）：

1. **重复触发会重复产生副作用吗？**
   - 这段代码被重复调用 / 用户重进页面 / 任务重跑时，会不会把同一个副作用做第二遍（重复发消息、重复扣款、重复写入）？

2. **有没有"已处理"的判断？**
   - 是否有 flag / 唯一键 / 去重 set 来识别"这条已经处理过"？

3. **"已处理"标记持久化了吗？**
   - 重启 / 换进程后，这个标记还认得吗？还是只存在内存里，一重启就失忆？

4. **失败的那条还能被重试吗？**
   - 幂等逻辑有没有"误杀"重试——把失败的也标记成已处理，导致永远不再重试？

5. **有没有对应的幂等测试？**
   - 是否有断言"跑两次，副作用次数不变"的测试？没有就补。

## 适用场景

- 批量任务、消息推送、支付/扣费、文件写入、外部 API 调用、定时任务、数据迁移、Webhook 处理。

## 输出

对每个命中场景，记录：场景位置(file:line) → 5 问的答案 → 风险等级 → 需补的逻辑/测试。
