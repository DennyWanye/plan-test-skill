# 调研方法论 — 毛选式问题解决

本文档是 phase-0（架构调研）、phase-1（最佳实践调研）、phase-2（迭代补充调研）的**共用调研纪律**。调研类步骤开始前先过一遍本清单；plan-challenger 会按这些条目反向执法。

## 六条方法

### 1. 没有调查，就没有发言权（《反对本本主义》）

- plan / 架构文档里的**每个论断必须附代码证据**（file:line 或命令输出），禁止"我认为应该是""一般来说是"。
- 没读过的代码不许下结论；来不及读的，明确标注为"未调查项"进入待闭环清单，不许写成结论。
- 调查像"十月怀胎"，解决问题像"一朝分娩"——调研没做够就动笔写方案，返工成本远大于调研成本。

### 2. 抓主要矛盾（《矛盾论》）

- 调研的第一步**不是罗列所有问题**，而是先回答：**这个需求里决定成败的核心问题是哪一个？**
- 把它写进 plan 的开头（"主要矛盾：……"），调研资源优先砸向它；次要问题跟着主要矛盾的解法走，不单独展开。
- 主要矛盾判断错了，后面全错——这一条允许在迭代中被挑战者推翻重来。

### 3. 具体问题具体分析（反对本本主义）

- 业界最佳实践**不能照搬**。每引用一个外部方案，必须写清：
  - 该实践成立的前提条件是什么；
  - 本项目的具体条件（规模、依赖、团队、存量代码）与之哪里相同、哪里不同；
  - 因此照用 / 改造后用 / 不用，改造点是什么。
- 只贴"业界都这么做"而无本项目适配分析的调研结论，视为未完成。

### 4. 解剖麻雀（典型调查）

- 与其对十个模块浅调研，不如**选一条典型调用链从入口挖到落库**，吃透一个，再把认识推广到同类。
- 选"麻雀"的标准：最能代表本次改动模式的那条链路（同类改动里最复杂或最常见的一个）。
- 解剖结论写成"这类改动的通用模式"，其余同类改动点引用该模式即可，不必逐个重复深挖。

### 5. 实践—认识—再实践（《实践论》）

- "读代码读不出来"的不确定项（运行时行为、性能、第三方库真实表现），**允许用小 spike 闭环**：写一段可丢弃的验证代码跑一下，把结论写回 plan，代码即弃。
- spike 是调研手段，不是实现的开始——不许把 spike 代码直接滚成实现。
- 迭代循环（挑战 → 优化 → 再挑战）本身就是"实践—认识—再实践"：每轮挑战暴露的认识盲区，用新一轮调研闭环，而不是用措辞圆过去。

### 6. 集中优势兵力，各个歼灭（不平均用力）

- 并行派子代理调研时，**多个代理集中攻坚最难的那一个不确定项**，各自从不同角度切入（读实现 / 读调用方 / 跑 spike），而不是每人平均分一块。
- 歼灭一个，再打下一个；一次并行只围一个歼灭目标，避免所有问题都推进 60% 却没有一个闭环。

## 调研产出自查

调研类步骤收尾前，逐条自问：

- [ ] 主要矛盾是否已明确写出，且调研深度与之匹配？
- [ ] 每个论断是否有 file:line 或命令输出证据？
- [ ] 外部最佳实践是否都有"本项目适配分析"？
- [ ] 是否至少解剖了一只麻雀（一条典型链路挖到底）？
- [ ] 剩余不确定项是否都有闭环手段（继续读 / spike / 问用户），而非模糊收尾？
